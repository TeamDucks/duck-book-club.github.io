Refactoring

  • Is conceived in Smalltalk circles.
  • Is making small changes to make source code easier to read without changing software outer behaviour.
  • Is not optimizing : optimizing make source code run faster.
  • Dev often changes between 2 hats: refactoring and adding new features.

Steps of refactoring

  • First step:
    Build solid set of tests
  • Rhythm of refactoring:
    test - small change - test - small change -...

Benefits of refactoring

  • Keep source code to design
  • Make source code easier to understand
  • Help find bugs
  • Help program faster
  • Refactoring is for software's tomorrow value(Kent Beck)

When to refactor

  • Rule of three
  • When adding function
  • When need to fix a bug
  • When doing code review

Refactoring and indirection

  • Software engineering is process of indirection.
  • Refactoring introduce more indirection into the code.
  • ...But indirection can pay for themself.

Refactoring and changing interfaces

  • Try to arrange old interfaces to call new ones, DO NOT COPY
  • Do not publish your interface prematurely

Refactoring and design

  • Flexible design form start is hard
  • Simple design and refactor later might work

Refactoring and optimization

  • Write first then optimize
  • Write and optimize at the same time
  • Write clear code first, then profile and optimize

Code smells

  • Duplicated code: extract method, substitute algorithm, extract class
  • Long method: look for comments, they 're candidates for extract method
  • Large class: Extract class, extract subclass
  • Long parameter list: replace parameter with method, introduce parameter object
  • Divergent change: difference types of common change point to same object - extract class
  • Shotgun surgery: one common change involves many places - move method, move field, inline class

Code smells

  • Feature envy: a method interested in other class more than its - move method, extract method(envy bits)
  • Data clumps: data chunking together - extract class, introduce parameter object, preserve whole object
  • Primitive obsession: tend of not using small classes
  • Switch statement: replace type code with sub-classes
  • Parallel inheritance hierarchy: same hierarchy repeats

Code smells

  • Lazy class: class that does not pay for itself - inline class, collapse hierarchy
  • Speculative Generality: 'we might need it someday' saying
  • Temporary Field: object with instance variable is set only in certain circumstances
  • Message chain: long chain of get... calls - hide delegate, move method
  • Middle man: delegating going too far - remove middle man, replace delegation with inheritance

Code smells

  • Inappropriate Intimacy: object spend too much time meddling with other's private data
  • Alternative Classes with Different Interfaces
  • Incomplete Library Class
  • Data Class
